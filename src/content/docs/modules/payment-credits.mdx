# 订阅支付与积分

商业化模块由 Creem 支付、订阅计划、积分发放和积分消费组成。订阅决定用户权益，积分账本记录每一次获得、消费、过期和人工调整。

## 实现了什么

- Free、Starter、Pro、Ultra 四档计划。
- Creem Checkout 创建订阅支付链接。
- Creem Webhook 同步订阅生命周期。
- 月度订阅积分发放。
- 一次性积分包配置。
- 注册奖励积分。
- 管理员人工充值积分。
- FIFO 批次扣减。
- 双重记账交易审计。
- 过期批次处理任务。

## Creem 配置

```bash
CREEM_API_KEY=creem_api_key_xxx
CREEM_WEBHOOK_SECRET=creem_webhook_secret_xxx

NEXT_PUBLIC_CREEM_PRICE_STARTER_MONTHLY=price_xxx
NEXT_PUBLIC_CREEM_PRICE_STARTER_YEARLY=price_xxx
NEXT_PUBLIC_CREEM_PRICE_PRO_MONTHLY=price_xxx
NEXT_PUBLIC_CREEM_PRICE_PRO_YEARLY=price_xxx
NEXT_PUBLIC_CREEM_PRICE_ULTRA_MONTHLY=price_xxx
NEXT_PUBLIC_CREEM_PRICE_ULTRA_YEARLY=price_xxx
```

在 Creem Dashboard 中创建产品和价格后，把价格 ID 填入对应变量。未配置价格 ID 的计划不应在生产中开放购买。

## 套餐配置

| 套餐 | 月付 | 年付 | 积分 | 文件上限 | 队列 |
| --- | --- | --- | --- | --- | --- |
| Free | 0 | 0 | 200，一次性 | 5MB | normal |
| Starter | 5 USD | 35 USD | 3,000/月 | 20MB | normal |
| Pro | 9 USD | 65 USD | 8,000/月 | 50MB | priority |
| Ultra | 15 USD | 109 USD | 16,000/月 | 100MB | highest |

展示价格在 `src/config/payment.ts`，权限限制在 `src/config/subscription-plan.ts`。调整套餐时必须同时更新这两个文件。

## 支付流程

1. 用户点击订阅按钮。
2. 前端提交 `priceId`。
3. `createCheckoutSession` 检查用户是否已有活跃订阅。
4. 系统根据 `priceId` 找到计划。
5. 调用 Creem 创建 Checkout。
6. 用户完成支付。
7. Creem Webhook 回调应用。
8. 应用写入或更新 `subscription` 表。
9. 成功后用户回到 `/dashboard?success=true`。

如果用户已有 `active`、`trialing` 或 `lifetime` 订阅，系统会阻止重复订阅。

## 取消订阅

`cancelSubscription` 会调用 Creem API 取消订阅，并将本地 `subscription.cancelAtPeriodEnd` 设为 `true`。Creem 当前没有类似 Stripe Customer Portal 的完整门户，前端应提示用户订阅管理方式或联系支持。

## 积分来源

| 来源 | 交易类型 | 批次来源 |
| --- | --- | --- |
| 注册奖励 | `registration_bonus` | `bonus` |
| 订阅月度发放 | `monthly_grant` | `subscription` |
| 一次性购买 | `purchase` | `purchase` |
| 管理员充值 | `admin_grant` | `bonus` |
| 退款 | `refund` | `refund` |
| 过期 | `expiration` | 扣减批次 |
| 消费 | `consumption` | 扣减批次 |

注册奖励默认 200 积分。积分过期策略当前为 `CREDITS_EXPIRY_DAYS = null`，表示永久有效。

## 积分包

一次性积分包定义在 `src/features/credits/config.ts`：

| ID | 名称 | 积分 | 价格 |
| --- | --- | --- | --- |
| `lite` | Lite | 3,000 | 3 USD |
| `standard` | Standard | 8,000 | 8 USD |
| `pro` | Pro | 20,000 | 18 USD |

如果开放积分包购买，需要确保支付回调能按 `purchase` 来源创建批次并记录交易。

## 双重记账设计

每次积分变动都写入 `credits_transaction`：

| 场景 | 借方 | 贷方 |
| --- | --- | --- |
| 发放注册奖励 | `SYSTEM:registration_bonus` | `WALLET:<userId>` |
| 管理员充值 | `ADMIN:<adminUserId>` | `WALLET:<userId>` |
| 消费积分 | `WALLET:<userId>` | `SERVICE:<serviceName>` |
| 积分过期 | `WALLET:<userId>` | `SYSTEM:expired` |

`credits_balance` 是预计算余额，用于快速读取；审计时应以 `credits_transaction` 和 `credits_batch` 为依据。

## FIFO 消费规则

消费积分时系统会：

1. 检查账户是否存在、是否冻结、余额是否足够。
2. 查询用户活跃批次。
3. 按过期时间和发放时间排序。
4. 从最早过期或最早发放的批次开始扣减。
5. 批次扣完后标记为 `consumed`。
6. 写入消费交易。
7. 更新 `credits_balance.balance` 和 `totalSpent`。

## 管理员人工充值

1. 管理员进入 `/admin/users`。
2. 找到目标用户。
3. 输入充值积分数和原因。
4. 系统调用 `adminGrantCreditsAction`。
5. 积分来源记录为 `bonus`，交易类型为 `admin_grant`。
6. 交易 metadata 记录管理员 ID 和原因。

单次人工充值上限为 100,000 积分。

## 风险控制

- 支付上线前必须验证 Creem Webhook 签名。
- 价格 ID 必须和套餐映射一致，避免错发权益。
- 订阅状态和积分发放应具备幂等性，防止重复 Webhook 重复发放。
- 手工充值必须填写原因，运营每周抽查大额发放。
- 积分消费失败时不要先扣批次后写交易，必须保持事务一致。
- 余额异常时先冻结账户，再核对交易和批次。
