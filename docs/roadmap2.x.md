# NextDevTpl 2.x 路线图

> 当前版本：`v2.2.0`
>
> 当前阶段：M3 项目生成器与预设（进行中）
>
> 历史路线图：[NextDevTpl 1.x 路线图](roadmap1.x.md)
>
> 最近更新时间：2026-07-23

## 项目目标

NextDevTpl 2.x 要成为一个可组合、可替换、可部署到多种平台的 SaaS
模板。仓库可以提供丰富能力，用户生成出来的项目只保留自己选择的源码、
依赖、环境变量、数据库表和文档。

## 核心原则

1. **生成结果干净**：未选择的功能不得进入最终项目。
2. **模块边界清楚**：业务模块只能依赖稳定内核，禁止互相穿透调用。
3. **适配器按需安装**：同类服务只生成用户选择的实现。
4. **部署目标显式选择**：服务器、Docker Compose、Vercel 和 Cloudflare
   Workers 各自拥有独立配置和验证流程。
5. **每种组合可验证**：核心组合必须经过构建、测试和部署冒烟检查。
6. **生成代码归用户所有**：线上运行不依赖脚手架，也不隐藏业务逻辑。

<a id="m0"></a>
## M0 - v2.0 基线

保留当前完整 SaaS 模板作为 2.x 改造起点，冻结主要功能清单和验证基线。
后续每次拆分都要证明现有认证、支付、积分、后台、工单、存储、国际化和
内容能力没有被意外破坏。

<a id="m1"></a>
## M1 - v2.1 模块内核解耦

拆清内核与业务模块的依赖方向，分离数据库 Schema、配置、路由、菜单和
翻译注册。这个版本以整理边界为主，尽量不增加新的业务功能，为后续按需
生成和多平台部署打基础。

<a id="m2"></a>
## M2 - v2.2 服务适配器体系

为支付、存储、邮件、AI、后台任务和限流建立小而稳定的接口，并提供首批
官方适配器。每个适配器使用同一套行为测试，供应商字段只能出现在对应实现
目录中。

首批计划支持：

| 能力 | 默认适配器 | 扩展适配器 |
|---|---|---|
| 支付 | Creem | Stripe |
| 存储 | S3 Compatible | Cloudflare R2 Binding |
| 邮件 | Resend | SMTP、Cloudflare Email |
| AI | OpenAI Compatible | Anthropic、Workers AI |
| 后台任务 | Inngest | Cloudflare Workflows |
| 限流 | Upstash Redis | Cloudflare Rate Limiting binding |

<a id="m3"></a>
## M3 - v2.3 项目生成器与预设

提供 `create-nextdevtpl`，让用户在创建项目时选择功能、适配器和部署目标。
生成器负责复制源码、安装依赖、组合 Schema、生成环境变量模板，并清除所有
未选模块；生成后的应用可以脱离生成器独立维护。

<a id="m4"></a>
## M4 - v2.4 常规部署目标

把服务器自部署、Docker Compose 和 Vercel 整理为正式部署预设。统一处理
构建、迁移、健康检查、日志和环境变量校验，同时允许数据库、Redis、对象
存储等可选服务独立启用。

<a id="m5"></a>
## M5 - v2.5 Cloudflare Workers

通过 OpenNext 支持 Cloudflare Workers，集中处理 Node.js 兼容、R2 Binding、
Cron、Queues 和 Workflows。首版数据库使用 Neon HTTP 或 Hyperdrive，D1 在
Schema 方言和迁移方案成熟后再评估。

<a id="m6"></a>
## M6 - v2.6 兼容性与稳定版

建立跨模块、跨适配器、跨部署目标的自动验证矩阵，补齐安装、卸载、替换、
升级和故障排查文档。这个版本作为 2.x 的长期稳定基线，为后续团队空间等
大型能力预留清晰扩展点。

## 计划中的仓库形态

```text
templates/base/               基础模板
recipes/                      可选功能与供应商适配器
packages/create-nextdevtpl/   项目生成器
examples/full/                完整 SaaS 演示项目
tests/compatibility/          组合与部署验证
```

目录调整必须分阶段完成。每一步都要保持当前模板可构建，禁止一次性搬迁导致
长时间不可用。

## 核心验证矩阵

| 预设 | 部署目标 | 关键组合 |
|---|---|---|
| `minimal` | 服务器 | PostgreSQL、基础认证、无可选业务模块 |
| `saas` | Docker Compose | PostgreSQL、Creem、积分、邮件、工单 |
| `saas` | Vercel | Neon、Creem、R2、Upstash、Inngest |
| `ai-saas` | Cloudflare Workers | Neon/Hyperdrive、R2、Workers AI、Workflows |

每个里程碑至少运行与其改动相关的单元测试、类型检查和构建。M4-M6 还必须
提供实际部署后的只读健康检查结果，只有配置文件或构建成功不能作为完成依据。

## 暂缓范围

- 团队空间、成员邀请和团队账单计划放到 3.x。
- D1 暂不作为 2.5 首发数据库，避免同时维护 PostgreSQL 与 SQLite 两套核心
  Schema。
- CMS、原生移动端和复杂工作流编辑器不进入 2.x 主线。
- 适配器市场暂缓，2.x 先把官方适配器契约和生成流程做稳。

## 完成定义

一个里程碑只有同时满足以下条件才能标记为“已完成”：

1. 对应模块文档已经反映实际代码和限制。
2. 核心业务逻辑具备单元测试，展示层不增加单元测试。
3. 格式化、静态分析、类型检查、测试和构建全部通过。
4. 涉及部署时，目标环境已经完成真实冒烟验证。
5. 改动已 commit 并推送，完成依据记录在模块文档中。

## 关键决策

| 日期 | 决策 | 原因 |
|---|---|---|
| 2026-07-23 | 2.x 优先处理模块解耦和代码纯净度 | 模板用户应只承担自己选择的代码和依赖 |
| 2026-07-23 | 适配器在生成阶段选定 | 避免最终应用长期携带未使用实现和运行时开关 |
| 2026-07-23 | Cloudflare Workers 排在常规部署之后 | Workers 支持依赖稳定模块边界和运行环境适配层 |
| 2026-07-23 | Workers 首版沿用 PostgreSQL | 先减少数据库方言差异，再单独评估 D1 |
