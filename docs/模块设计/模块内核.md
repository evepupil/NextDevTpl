# 模块内核

> 模块定位：定义基础内核、业务模块边界和依赖方向
>
> 对应代码：计划覆盖 `src/core/`、`src/features/`、`src/db/schema/`
>
> 所属里程碑：[M1 - v2.1 模块内核解耦](../roadmap2.x.md#m1)
>
> 当前状态：未开始
>
> 最近更新时间：2026-07-23

## 职责与边界

内核只提供配置、身份上下文、日志、错误模型、数据库连接和模块注册等稳定
能力。支付、积分、AI、工单等业务模块依赖内核，业务模块之间通过明确接口
协作，禁止直接读取对方内部文件。

## 结构与数据流

计划把调用方向固定为：页面或 API -> 业务用例 -> 仓储或服务接口 -> 具体
实现。数据库 Schema、菜单、翻译和环境变量声明随模块归档，最终由构建或
生成阶段组合。

## 关键决策

- 保持 Server Components 和 Server Actions 的现有使用方式。
- 只抽象跨模块或跨供应商的稳定边界。
- 模块公开能力统一从 `index.ts` 导出。
- 数据库 Schema 按模块拆分，迁移仍由一个 Drizzle 配置统一生成。

## 当前实现

现有代码已按 `src/features/` 分区，但 Schema、导航、翻译和部分配置仍集中，
部分模块会直接引用其他模块内部实现。M1 开始时需要先完成依赖盘点。

## 验证方式

- 为核心用例、仓储和状态计算保留或补充单元测试。
- 增加模块依赖检查，阻止跨模块深层导入。
- 分别移除 AI、工单、支付等可选模块后执行 typecheck 和 build。
- 运行格式化、lint、typecheck、测试和生产构建。

## 待扩展项

- 设计模块清单文件及 Schema 聚合方式。
- 评估现有路由是否需要生成阶段注入。
