# 模块内核

> 模块定位：定义基础内核、业务模块边界和依赖方向
>
> 对应代码：`src/core/modules/`、`src/modules/`、`src/features/`、
> `src/db/schema/`、`src/config/nav.ts`
>
> 所属里程碑：[M1 - v2.1 模块内核解耦](../roadmap2.x.md#m1)
>
> 当前状态：进行中
>
> 最近更新时间：2026-07-23

## 职责与边界

内核只定义稳定的数据契约和校验规则，不读取任何业务模块。认证、支付、积分、
工单等业务能力各自维护公开入口和模块声明；组合层负责选择并注册模块，页面、
API 和其他模块只能通过公开入口使用它们。

M1 负责建立可执行的模块边界和组合信息，不改变现有产品功能，也不引入新的
供应商抽象。支付、存储、邮件等供应商适配将在 M2 处理，源码生成和物理删除
将在 M3 使用本阶段产出的模块清单。

## 结构与数据流

```text
src/core/modules/contract.ts       模块声明类型与纯校验函数
          ^
          | 只依赖稳定契约
src/features/*/manifest.ts         各模块声明依赖、路由、菜单、翻译和 Schema
          ^
          | 由组合入口显式注册
src/modules/registry.ts            完整模板的模块注册表
          |
          +-> src/config/nav.ts    组合 Dashboard / Admin 菜单
          +-> 架构测试             检查依赖、公开入口和翻译命名空间

src/db/schema/*.ts                 按模块维护表结构
src/db/schema/index.ts             Drizzle 使用的统一 Schema 出口
```

页面或 API 的调用方向固定为：入口 -> 模块公开 API -> 模块内部实现 -> 数据库
或外部服务。模块内部可以深层引用自身文件；跨模块调用和 `src/app/` 等外部入口
必须使用 `@/features/<模块>`。

## 模块声明

每个模块通过纯数据声明以下信息：

- `id`、类型和说明；
- 直接依赖的其他模块；
- 拥有的页面与 API 路由；
- Dashboard 或 Admin 菜单贡献；
- 使用的翻译顶层命名空间；
- 需要聚合的数据库 Schema 分组。

注册表校验模块 ID、依赖存在性、循环依赖、重复路由和重复菜单地址。M3 可以
基于同一份声明计算选中模块的依赖闭包，并删除未选模块拥有的路由和 Schema。

## 关键决策

- 保持 Server Components、Server Actions 和 App Router 的现有使用方式。
- `src/core/` 永远不导入 `src/features/`；完整模板的组合入口放在
  `src/modules/`。
- 模块公开能力统一从 `index.ts` 导出，跨模块深层导入由测试直接拦截。
- Schema 按认证、订阅、积分、邮件和工单拆分，表名、字段和迁移结果保持不变。
- 菜单由模块声明组合；图标映射留在展示配置层，核心契约只保存图标名称。
- 翻译文件本阶段继续保留中英文总文件，模块声明负责登记命名空间并检查两种
  语言都存在。M3 生成项目时再按登记结果裁剪 JSON。
- App Router 路由暂不搬动，模块声明记录文件归属，供 M3 生成器执行删除。

## 当前实现审计

- 15 个 feature 都已有顶层 `index.ts`，但页面和跨模块代码存在大量深层导入。
- `src/db/schema.ts` 共 517 行，混合认证、订阅、积分、邮件和工单五组表。
- Dashboard、Admin 和 Marketing 菜单集中在 `src/config/nav.ts`，可选模块入口
  无法独立移除。
- 中英文翻译集中在 `messages/<locale>.json`，没有模块所有权登记。
- 已确认的直接业务依赖包括：积分 -> 支付、Dashboard -> 积分/订阅、Settings
  -> 积分/支付/存储/订阅、Support -> 积分，以及多个 UI 模块 -> Shared。

## 实施计划

1. 增加模块契约、完整模板注册表和注册表单元测试。
2. 为全部 feature 增加模块声明，并让 Dashboard/Admin 菜单从声明组合。
3. 拆分 Schema 并切换 Drizzle、业务模块和测试到明确的 Schema 入口。
4. 补齐各 feature 的公开导出，清除跨模块和入口层深层导入。
5. 增加源码边界、依赖声明、路由归属和翻译命名空间检查。
6. 验证最小模块集合的依赖闭包，再运行完整项目门禁。

## 当前实现

M1 已完成现状审计，模块契约与代码拆分正在实施。现有运行行为仍以 v2.0
基线为准。

## 验证方式

- 模块注册表测试：ID、依赖、循环、路由、菜单、Schema 和翻译登记。
- 源码边界测试：禁止跨模块深层导入，禁止依赖未声明的业务模块。
- 最小集合测试：验证基础模块的依赖闭包不包含可选业务模块。
- Schema 回归：Drizzle 聚合出口保持原有表和类型，并运行现有数据库测试。
- 项目门禁：format、lint、typecheck、276 项测试和生产构建。

## 待扩展项

- M2 将支付、邮件、存储、AI、后台任务和限流的供应商实现放到适配器后面。
- M3 根据模块声明生成实际源码、路由、Schema、环境变量和翻译子集。
- M6 在 CI 中扩展多种模块组合的真实构建矩阵。
