> 模块定位：支持服务器、Docker Compose 和 Vercel 的正式部署预设
>
> 对应代码：`src/lib/health/`、`src/app/api/health/route.ts`、`deploy/`、
> `Dockerfile`、`compose.yaml`、`vercel.json`、`packages/create-nextdevtpl/`
>
> 所属里程碑：[M4 - v2.4 常规部署目标](../roadmap2.x.md#m4)
>
> 当前状态：阻塞
>
> 最近更新时间：2026-07-23

## 职责与边界

部署预设负责构建、启动、迁移、健康检查和平台环境变量映射。业务模块不得在
内部判断 Vercel、Docker 或服务器环境，平台差异集中在部署配置和适配器层。

M4 覆盖常规 Node.js 运行环境。Cloudflare Workers 的 OpenNext 产物、Binding
注入和部署验证归 M5，避免两种运行模型互相牵连。

## 结构与数据流

```text
部署目标选择
    |
    v
create-nextdevtpl 复制统一应用模板
    |
    +-- server  -> standalone 产物 + systemd/启动说明
    +-- docker  -> 多阶段镜像 + Compose 应用/迁移/PostgreSQL
    +-- vercel  -> Vercel 配置 + 托管服务接入说明
    |
    v
GET /api/health?mode=live   只检查进程是否能响应
GET /api/health             校验运行配置并执行数据库只读查询
```

生成器先复制同一份应用模板，再根据 `target` 删除其他平台文件。健康检查和
数据库查询由应用内的共享模块实现，部署脚本只负责请求接口，不复制判断逻辑。

## 关键决策

- Next.js 使用 `standalone` 产物，服务器与容器运行相同的 `server.js`。
- 生产构建固定使用 webpack。Next 16 Turbopack 在 Windows 生成 standalone 时会
  对含 `node:` 的追踪文件报 `EINVAL` 且仍返回成功，容易留下不完整产物。
- 健康接口默认执行 `SELECT 1` 等价的只读查询，成功返回 200，配置或数据库失败
  返回 503；响应不包含连接串、密钥、账户名、异常消息或调用栈。
- `mode=live` 仅用于进程存活判断，不能代替默认的就绪检查。
- 服务器预设使用普通 Node.js 进程和 systemd 示例，不强制依赖全局 PM2。
- Docker Compose 使用 named volume 保存 PostgreSQL 数据，并用一次性迁移容器
  阻挡应用启动；数据库数据不得映射到宿主项目目录。
- Vercel 预设使用 Node.js Serverless 运行时，数据库迁移作为发布前独立步骤，
  避免多个构建实例并发修改 Schema。
- 对象存储、队列、Redis 等继续使用 M2 的外部服务适配器。M4 不在 Compose 中
  私自增加与所选适配器不一致的本地替代服务。

## 当前实现

`deploy-build.bat` 与 `start-prod.sh` 已由正式预设替换，当前实现包括：

1. `/api/health` 默认校验必要配置并执行数据库只读查询，`mode=live` 只检查
   进程存活；失败统一返回 503，不向响应写入内部错误详情。
2. 服务器预设生成 webpack standalone 产物，提供迁移、启动、systemd 示例、
   健康检查脚本和操作文档。
3. Docker 预设使用多阶段镜像、非 root 应用进程、一次性迁移容器和 PostgreSQL
   named volume；应用只在数据库健康且迁移成功后启动。
4. Vercel 预设提供配置和发布说明，数据库迁移留在发布前的受控单实例步骤。
5. 生成器按 `server`、`docker`、`vercel`、`cloudflare` 裁剪部署文件，并在安装后
   按裁剪后的 Schema 生成初始迁移。
6. `ws` 与 Neon 驱动作为 standalone 外置生产依赖，避免运行产物缺失 WebSocket
   实现。

## 验证方式

- 根项目已通过 format、lint、typecheck、25 个测试文件共 323 个用例和 webpack
  standalone 构建；生产产物的 `/api/health` 返回 `healthy`，数据库检查为 `pass`。
- `minimal/server`、`saas/vercel`、`saas/docker` 三组生成项目均通过依赖安装、
  初始迁移生成、lint、typecheck、test、build 和 standalone 健康检查。
- Docker 已完成真实 Compose 冒烟：数据库和应用容器均为 healthy，迁移容器
  退出码为 0；重建后迁移记录仍为 2，证明 named volume 数据持续存在。
- Vercel 目标已通过生成、生产构建和本地 standalone 健康检查。远端冒烟只发布
  到用户确认的测试项目，结果必须来自 preview 的 `/api/health`。

## 当前阻塞

M4 还缺 Vercel preview 的真实部署验证。需要用户提供或确认一个 Vercel 测试
项目，在项目中配置 `.env.example` 要求的测试环境变量，并允许执行 preview
部署。完成远端 `/api/health` 检查后才能把 M4 标记为已完成。

## 待扩展项

  和数据库，避免外部服务短暂波动造成所有实例被同时摘除。
- M5 增加 Cloudflare Workers 对应的健康探测与部署产物。
