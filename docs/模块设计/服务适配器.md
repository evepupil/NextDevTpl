# 服务适配器

> 模块定位：隔离支付、存储、邮件、AI、任务和限流供应商差异
>
> 对应代码：`src/core/services/`、`src/adapters/`、`src/services/`
>
> 所属里程碑：[M2 - v2.2 服务适配器体系](../roadmap2.x.md#m2)
>
> 当前状态：已完成
>
> 最近更新时间：2026-07-23

## 职责与边界

业务层只使用项目定义的能力接口。供应商 SDK、字段映射、签名校验和错误码
处理全部放在适配器内部。适配器不能改变积分、订阅等核心业务规则，也不能
读取其他业务模块的数据。

M2 覆盖支付、对象存储、事务邮件、文本生成、后台任务和请求限流。页面、API、
Server Action 与业务模块只能调用 `src/services/` 的装配结果或
`src/core/services/` 的稳定类型。

## 结构与数据流

```text
src/core/services/*          稳定契约、能力声明、统一错误
          ^
          | implements
src/adapters/<能力>/<供应商>  SDK 调用、字段转换、签名验证
          ^
          | 完整模板选择默认实现
src/services/*               组合入口
          ^
          | 只暴露项目自己的类型
业务模块、API、Middleware
```

业务用例调用能力接口，生成阶段为每项能力绑定一个实现。外部事件先由支付适配器
转换成内部事件，再交给 webhook 用例处理。内部表结构不保存无必要的供应商
对象。每个适配器显式声明能力差异，例如原生 R2 Binding 不提供 S3 预签名 URL，
Cloudflare Rate Limiting Binding 不返回精确剩余额度。

## 关键决策

- 最终项目只保留所选适配器及其依赖。
- 每类适配器拥有统一契约测试和错误模型。
- webhook 幂等、重试和审计属于公共业务能力。
- OpenAI Compatible 作为 AI 默认接口，特殊能力通过显式扩展暴露。
- 完整模板保持现有默认供应商行为；M3 在生成阶段替换 `src/services/` 的装配
  文件并删除未选适配器。
- Cloudflare 原生绑定通过构造参数注入。M5 生成 `wrangler types` 后将真实
  `Env` binding 传给适配器，M2 不提前引入 Worker 全局类型。
- Cloudflare Email 只用于事务邮件。启用时必须先开通发件域名并配置
  `send_email` binding，同时发送 HTML 和纯文本正文。

## 当前实现

六类能力已经完成契约、供应商实现和默认装配：

| 能力 | 默认适配器 | 扩展适配器 | 已声明的主要差异 |
|---|---|---|---|
| 支付 | Creem | Stripe | 结账、订阅和 webhook 统一为内部支付对象 |
| 存储 | S3 Compatible | R2 Binding | R2 Binding 支持直接读写，不支持预签名 URL |
| 邮件 | Resend | SMTP、Cloudflare Email | Cloudflare Email 必须同时发送 HTML 和纯文本 |
| AI | OpenAI Compatible | Anthropic、Workers AI | JSON 输出能力显式声明，不支持时返回 `unsupported` |
| 后台任务 | Inngest | Cloudflare Workflows | 统一使用任务名、JSON 载荷和可选幂等 ID |
| 限流 | Upstash Redis | Cloudflare Rate Limiting | Cloudflare Binding 不返回剩余量和重置时间 |

业务迁移结果：

- 支付 Action、账户删除和 Creem webhook 已改用 `paymentService`，Creem/Stripe
  字段与签名校验留在支付适配器内；
- 头像上传、通用预签名上传和文件代理共用 `storageService`，删除第二套 S3
  客户端；
- React Email 模板先渲染为 HTML 和纯文本，再交给 `mailService`；
- AI 兼容调用面保留 `chatCompletion()`，内部通过 `aiService` 完成；
- Inngest 实现迁入任务适配器，业务派发使用 `jobService`；
- Middleware 与匿名配额共用限流装配，未配置 Upstash 时使用显式 noop 降级。

`src/adapters/registry.ts` 登记每个适配器的源码、依赖、环境变量、Binding 和
运行时。`createServiceAdapterSelection()` 已能输出 M3 裁剪项目所需的完整清单。
选择类型按能力分类，无法把邮件适配器误配到支付等其他能力。

## 验证方式

- 所有正式适配器通过统一描述符和契约操作检查；使用注入的测试替身验证
  Creem/Stripe webhook、R2、Cloudflare Email、Anthropic、Workers AI、
  Workflows 和 Cloudflare Rate Limiting 的字段映射与能力差异。
- 架构测试扫描生产源码，供应商 SDK 只能在 `src/adapters/` 中导入；供应商实现
  只能由 `src/services/` 和 Inngest 的协议入口路由装配。
- 统一错误会保留供应商、错误码和重试属性，并从消息中清理已知密钥。
- 2026-07-23 门禁：`pnpm lint`、`pnpm typecheck`、`pnpm test:run`、
  `pnpm build` 全部通过；完整测试为 21 个文件、303 个用例。
- 实现提交：`4083553`（`feat: establish service adapter system`）。
- M2 只验证本地契约和注入式替身。真实供应商冒烟验证随对应部署预设在 M4/M5
  完成，当前没有把未部署结果记为真实可用。
