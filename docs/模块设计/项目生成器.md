# 项目生成器

> 模块定位：按用户选择生成纯净、可独立维护的 NextDevTpl 项目
>
> 对应代码：`packages/create-nextdevtpl/`、`templates/`、`recipes/`、
> `scripts/verify-generated-projects.mjs`
>
> 所属里程碑：[M3 - v2.3 项目生成器与预设](../roadmap2.x.md#m3)
>
> 当前状态：已完成
>
> 最近更新时间：2026-07-23

## 职责与边界

生成器负责收集选择、解析依赖、复制文件和输出配置。生成器不参与线上运行，
也不替用户维护生成后的代码。

## 结构与数据流

```text
CLI 参数或交互选择
        |
        v
recipes/catalog.json       模块、适配器、预设和部署目标清单
        |
        v
选择解析器                 依赖闭包、冲突和运行时兼容检查
        |
        v
模板复制 -> 路由/模块裁剪 -> 注册表/Schema/翻译/服务装配重建
        |
        v
独立项目 + nextdevtpl.generated.json + 可选 pnpm install
```

仓库开发时直接使用当前源码作为模板来源。发布 `create-nextdevtpl` 前，`prepack`
脚本根据 `templates/base/manifest.json` 组装只读模板快照和 recipe 清单，避免手工
维护第二套源码。生成结果不包含生成器包、roadmap、测试数据库配置和完整模板的
迁移历史。

## 关键决策

- 提供 `minimal`、`saas`、`ai-saas` 三个官方预设。
- 支持无交互参数，方便 CI 和自动化使用。
- 使用 JSON、AST 或框架配置 API 修改结构化文件，避免脆弱的字符串替换。
- 生成结果不依赖 `create-nextdevtpl` 包。
- 目标目录必须为空或不存在；生成失败只回滚本次新建的精确目录。
- M3 只记录部署目标和 Binding 需求。Docker/Vercel 配置由 M4 生成，Cloudflare
  Binding 类型与运行时注入由 M5 完成。
- 完整模板继续在仓库根目录开发；发布包通过自动快照保持模板与源码同源。

## 当前实现

`create-nextdevtpl` 已作为独立 workspace 包实现。CLI 支持官方预设和自定义
模块，服务适配器与部署目标均可通过参数覆盖：

```text
create-nextdevtpl <directory>
  --preset minimal|saas|ai-saas|custom
  --modules auth,dashboard,...
  --target server|docker|vercel|cloudflare
  --payment/--storage/--mail/--ai/--jobs/--rate-limit <adapter>
  --no-install --yes
```

选择解析器会计算模块依赖闭包，校验适配器能力分类和 Node/Worker 运行时兼容，
并拒绝非空目标目录。生成失败时只删除本次新建的目标目录，已有空目录不会被
整体删除。

生成流程已经覆盖以下内容：

- 删除未选 feature、路由目录、Schema 文件、适配器源码和供应商 SDK；
- 重建模块注册表、Schema 汇总、翻译、`.env.example`、服务装配和包信息；
- 按选择重建 Locale、Dashboard 与 sitemap 组合入口；
- 支付 webhook 统一为 `/api/webhooks/payment`，兼容 Creem 和 Stripe 签名头；
- 生成 `nextdevtpl.generated.json`，记录模块、适配器、目标和 Binding 需求；
- 生成轻量 smoke test，不复制依赖完整测试数据库的仓库测试集；
- 内容型项目安装后自动运行 `fumadocs-mdx`，lockfile 保证依赖版本可复现。

`prepack` 会从当前仓库自动组装发布快照。快照保留 `.gitignore`、
`next-env.d.ts` 和 lockfile，不包含本地环境文件、缓存、依赖目录或完整测试集。
生成阶段会用 YAML 解析器同步 lockfile 根 importer 与裁剪后的 `package.json`，
所以生成项目可以直接执行冻结安装。生成项目不依赖生成器包，可以独立维护。

Cloudflare 组合目前输出所需 Binding，并在装配点给出未注入错误。实际 Worker
`Env` 注入、OpenNext 配置和部署验证属于 M5。

## 验证方式

- 在临时目录生成所有官方预设并安装依赖。
- 每个预设运行 lint、typecheck、测试和 build。
- 检查未选模块的文件、依赖、环境变量和数据库表全部缺席。
- Windows、Linux 和 CI 环境使用相同输入生成一致结果。
- 打包后检查 npm tarball 包含 CLI、模板快照和 recipe，不包含本地密钥或缓存。
- 2026-07-23 根门禁：format、lint、typecheck、build 全部通过；完整测试为
  25 个文件、318 个用例全部通过。
- `minimal`、`saas`、`ai-saas` 和一组 SMTP + S3 自定义组合均在独立临时目录
  完成安装、lint、typecheck、smoke test 和 build。
- npm tarball 与打包后 CLI 已验证；敏感或缓存条目计数为 0。
- 2026-07-24 M6 验证发现并修复裁剪后 lockfile 仍声明已删除依赖的问题；
  `minimal-server` 已通过 `pnpm install --frozen-lockfile`。
- 实现提交：`8c4e71f`（`feat: add composable project generator`）。
