# 兼容性与稳定性

> 模块定位：验证模块、适配器和部署目标的受支持组合
>
> 对应代码：`tests/compatibility/`、`scripts/verify-compatibility.mjs`、
> `scripts/compatibility/`、`.github/workflows/compatibility.yml`
>
> 所属里程碑：[M6 - v2.6 兼容性与稳定版](../roadmap2.x.md#m6)
>
> 当前状态：进行中
>
> 最近更新时间：2026-07-24

## 职责与边界

本模块维护官方支持矩阵、自动验证和发布准入条件。它验证已有能力，不承载新
业务功能，也不把所有理论组合都列为官方支持。

## 结构与数据流

```text
tests/compatibility/matrix.json
              |
              v
生成独立项目 -> 目标结构检查 -> 冻结安装 -> 迁移生成
              |                         |
              v                         v
       lint / typecheck / test -> 目标构建 -> 本地健康检查
              |
              v
 artifacts/compatibility/<case>.json -> CI 单组制品 -> 发布判断
```

CI 将四组组合拆成独立任务。每个任务只写自己的 JSON 报告，失败时保留阶段、
脱敏后的末尾日志和临时豁免信息。涉及平台的核心组合还要部署到测试环境，再
通过只读冒烟检查确认可访问性。

## 关键决策

- 官方矩阵保持少而明确，新增组合必须承担长期验证成本。
- 核心业务逻辑使用单元与集成测试，纯展示层依赖人工和截图检查。
- Release 必须关联通过的矩阵结果和已知限制。
- 失败日志必须隐藏密钥和用户数据。
- 依赖安装使用生成后同步过的 lockfile 和 `--frozen-lockfile`，防止 CI 静默
  改动依赖版本。
- 网络相关的安装阶段允许自动重试一次；代码、类型、测试和构建失败不自动
  重试，避免掩盖确定性问题。
- 临时豁免只能写入 `exemptions.json`，必须指定组合、阶段、原因、跟踪问题和
  到期时间。过期豁免会直接阻塞门禁。
- GitHub 仓库公开，当前矩阵使用 GitHub 托管 runner，不产生 Actions 分钟费用；
  真实外部服务测试仍按需控制资源生命周期。

## 当前实现

已经建立版本化的四组官方矩阵：

| 组合 | 预设 | 目标 | 主要验证 |
|---|---|---|---|
| `minimal-server` | `minimal` | Server | 独立构建、standalone 启动、数据库健康检查 |
| `saas-docker` | `saas` | Docker Compose | SaaS 构建、Compose 插值和服务依赖、健康检查 |
| `saas-vercel` | `saas` | Vercel | Vercel 配置、SaaS 构建和健康检查 |
| `ai-saas-cloudflare` | `ai-saas` | Cloudflare | OpenNext 构建、Worker 产物和 gzip 免费额度上限 |

结构测试由矩阵驱动，检查必需文件、禁止残留文件、模块移除和适配器替换。
已有 `src/test/adapters/` 契约测试继续覆盖六类能力，矩阵负责证明所选实现能在
最终生成项目里安装和构建。

生成器现在同步 `package.json` 与 lockfile 根 importer。未选供应商依赖会从
冻结安装输入中移除，已选依赖继续使用仓库锁定版本。

`.github/workflows/compatibility.yml` 在 PR、`master` 推送和手工触发时先运行整仓
质量门禁，再并行执行四组生成项目。每组报告即使失败也会上传，便于定位。
CI 为质量门禁和生成项目分别启动临时 PostgreSQL 17；测试工具使用 Node `pg`
协议，所以同一套用例可以连接本地 PostgreSQL 或 `.env.test` 指向的 Neon 分支。

## 2.x 兼容承诺

- Node.js 支持 24 主版本，`engines` 使用 `>=24 <25`；仓库开发和 CI 继续通过
  `.node-version` 固定已验证的补丁版本。
- 2.x 小版本保持生成清单格式、官方适配器接口、环境变量名称和四个部署目标可用。
- 生成代码归用户所有，不提供覆盖式升级。升级时按同一选择重新生成对照项目，再
  审查并分批合并差异。
- 删除或重命名公开接口、改变持久化数据语义的改动进入下一个主版本。安全修复和
  平台强制变更允许在 2.x 落地，但 Release 必须提供迁移和回退步骤。
