# NextDevTpl 2.x 升级指南

> 适用范围：由 `create-nextdevtpl` 生成并已脱离生成器独立维护的项目
>
> 稳定性设计：[M6 - 兼容性与稳定版](模块设计/兼容性与稳定性.md)

生成后的代码归项目所有。2.x 当前不提供覆盖式升级命令，升级采用“同选择重新
生成、审查差异、分批合并”的方式，避免覆盖项目自己的业务改动。

## 升级前

1. 提交或暂存当前项目改动，确认可以回退。
2. 备份数据库，并记录当前应用版本和迁移编号。
3. 保存 `nextdevtpl.generated.json`。它记录原预设、模块、适配器、部署目标和
   catalog 版本，不包含密钥。
4. 运行当前版本的 lint、类型检查、测试、构建和健康检查，先得到可比较基线。

## 生成对照项目

使用新版本生成器，在临时目录用原来的选择生成新项目。优先根据
`nextdevtpl.generated.json` 还原参数；项目后来手工增加的模块单独记录。

```bash
pnpm dlx create-nextdevtpl@<目标版本> ../nextdevtpl-upgrade \
  --preset <原预设> \
  --target <原目标> \
  --no-install --yes
```

先比较以下内容：

- `package.json`、`pnpm-lock.yaml` 和 Node/pnpm 版本；
- `drizzle/` 与 `src/db/schema/`，确认迁移只向前追加；
- `src/core/`、`src/adapters/` 和 `src/services/` 的契约变化；
- 部署配置、环境变量示例和 `nextdevtpl.generated.json`；
- 自己改过的页面、业务模块和翻译。

## 合并与验证

按“工具链和依赖、内核和适配器、业务模块、部署配置”的顺序分批合并。每批都
运行相关单测和类型检查。数据库迁移先在测试数据库执行，禁止删除或改写已经在
生产运行过的迁移。

最终至少执行：

```bash
pnpm install --frozen-lockfile
pnpm lint
pnpm typecheck
pnpm test:run
pnpm build
```

部署到 Preview 或临时环境后，再验证 `/api/health`、登录、数据库以及所选支付、
存储、邮件、AI 和后台任务能力。确认通过后再升级生产。

## 回退

代码回退到升级前提交。数据库只使用单独审核过的向前修复迁移，禁止直接回滚
已经写入生产数据的迁移。外部服务配置按变更记录逐项恢复。
