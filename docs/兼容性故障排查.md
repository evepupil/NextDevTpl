# 兼容性矩阵故障排查

> 对应模块：[M6 - 兼容性与稳定版](模块设计/兼容性与稳定性.md)

## 本地复现

先运行结构检查，确认生成选择和部署文件：

```bash
pnpm verify:compatibility:structure --case minimal-server
```

再运行完整单组检查：

```bash
pnpm verify:compatibility --case minimal-server
```

报告位于 `artifacts/compatibility/<case>.json`。报告只记录阶段状态、时间、产物
大小和失败时经过脱敏的末尾日志，不记录环境变量原值。

## 阶段含义

| 阶段 | 常见原因 | 先检查 |
|---|---|---|
| `generate` | recipe、模板清单或目标目录错误 | 生成器单测和 catalog |
| `target-config` | 目标文件缺失或残留其他目标文件 | matrix 的必需/禁止路径 |
| `deployment-config` | Compose 插值或平台配置错误 | 临时环境值、Compose 结构 |
| `install` | lockfile 不一致或网络错误 | `package.json` 与根 importer |
| `migration` | Schema 引用被裁剪或迁移生成失败 | 选中模块的 Schema 汇总 |
| `lint` / `typecheck` | 裁剪后留下悬空导入 | 首个报错文件和所属模块 |
| `test` | 生成清单或核心行为变化 | 失败断言与 `nextdevtpl.generated.json` |
| `build` | 运行时边界、环境变量或平台产物错误 | 目标构建命令的末尾日志 |
| `health` | standalone 启动或数据库不可用 | `/api/health` 和测试数据库 |

安装阶段会自动重试一次。其余阶段需要修复根因后重跑。

## Windows 与 WSL

Windows 负责生成、依赖安装、类型、测试和 Next.js 构建。Docker Compose 配置
检查会把临时路径转换为 `/mnt/c/...`，再调用 `Ubuntu-24.04` 中的 Docker。
Cloudflare OpenNext 的 Linux 专项复现也应从 WSL 运行。

如果 WSL 在 `/mnt/c` 下执行 `pnpm install` 时出现 `EACCES rename`，把生成项目
复制到 WSL 的 `/tmp` 独立目录再安装和构建。不要让 Windows 与 WSL 轮流复用同一
个 `node_modules`。

## Cloudflare 线上 500

先检查 Workers Logs 中第一条运行时堆栈。若堆栈包含 `SonicBoom`、`node:fs.write`
或 `validateWriteArgs`，说明生成结果错误保留了 Pino。当前 2.x 生成器会为
Cloudflare 目标输出专用 `console` logger，并从依赖中移除 `pino` 和
`pino-pretty`；重新生成项目后确认：

```bash
pnpm install --frozen-lockfile
pnpm cf:build
pnpm exec wrangler deploy --dry-run --minify
```

如果 `/api/health` 返回 `unhealthy` 而 Worker 没有异常，再分别检查配置校验和 Neon
只读探针。日志和诊断接口只能记录变量是否存在，禁止返回变量原值。

## 临时豁免

只有已经建立跟踪问题、影响范围明确且不阻塞核心流程时，才能修改
`tests/compatibility/exemptions.json`。每条必须包含 `caseId`、`stage`、`reason`、
`issue` 和 `expiresAt`。到期时间必须短于计划修复日期；过期记录会直接让门禁
失败。涉及认证、数据完整性、密钥泄露或无法启动的问题禁止豁免。
